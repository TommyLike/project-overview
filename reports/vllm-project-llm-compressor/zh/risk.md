# 风险评估 — LLM Compressor

> **来源**: https://github.com/vllm-project/llm-compressor | **分析日期**: 2026-02-18

## 整体风险：🟡 中等（可控）

项目由强力企业背书、积极维护，但有三个因素值得关注：v1 版本前的 API 不稳定性、缺乏正式的安全漏洞披露流程，以及与 compressed-tensors 格式的深度耦合。

## 破坏性变更风险

| 信号 | 评估 |
|------|------|
| 当前版本 | v0.9.x — **v1 版本前** |
| CHANGELOG | 不存在——无正式变更日志 |
| 语义化版本管控 | 次要版本（0.7 → 0.8 → 0.9）已引入新 API 和弃用项 |
| 迁移指南 | 未发现专门的迁移文档；README "What's New" 章节覆盖主要亮点 |
| API 接口稳定性 | `oneshot()` 入口点稳定；modifier 配置 schema 随次要版本演进 |

**评估**：🟡 **中等破坏性变更风险。** 该库明确处于 1.0 之前，依据语义化版本规范，不提供任何 API 稳定性保证。实际情况：
- `oneshot()` / `model_free_ptq()` 入口点稳定，变化保守。
- Modifier 配置 schema（YAML recipe）和内部 modifier API 变化较为频繁。
- `dispatch_for_generation` → `dispatch_model` 的弃用（2026 年 2 月）表明便捷 API 在次要版本间会带着弃用警告进行重命名。

**缓解措施**：在生产环境中固定次要版本（`llmcompressor~=0.9.0`），升级前充分测试。

## 生态系统耦合风险

LLM Compressor 特有的风险，其他框架不具备：

- **compressed-tensors 格式**：输出格式依赖 vLLM Project 的同名库（`compressed-tensors==0.13.0`，固定严格）。若 vLLM 放弃该格式，LLM Compressor 需同步更新。鉴于 Red Hat AI 同时掌控两个项目，此风险极低。
- **Transformers 兼容性**：`transformers>=4.56.1,<=4.57.6` 的固定范围（上下均有限制）极为严格。README 中可见一个关于 `transformers 4.57.3` 不兼容的开放 Issue——这表明 Transformers 快速迭代带来的依赖风险不容忽视。各团队应确保使用支持的 Transformers 版本。
- **要求 Python ≥ 3.10**：新项目无影响，但仍在 Python 3.9 的团队需升级。
- **仅支持 Linux**：`setup.py` 分类器标注为 `Operating System :: POSIX :: Linux`，macOS 和 Windows 不在官方支持范围内。
